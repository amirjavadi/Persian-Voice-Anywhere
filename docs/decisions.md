# ثبت تصمیم‌های معماری (ADR)

هر تصمیم مهم اینجا ثبت می‌شود: زمینه، تصمیم، دلیل و پیامدها. تغییر یک تصمیم = ADR
جدید که ADR قبلی را «جایگزین» می‌کند. وضعیت‌ها: پیشنهادی / پذیرفته / جایگزین‌شده.

---

## ADR-0001 — پلتفرم و زبان: .NET 10 + C# 14

**وضعیت:** پذیرفته · ۱۴۰۵/۰۴/۳۱ (2026-07-22)

**زمینه:** محصول یک اپ دسکتاپ ویندوز، پرتابل و کم‌مصرف است. مالک گفت زبان مهم نیست،
«بهترین انتخاب» و SDK نسخه‌ی ۱۰ روی سیستم نصب است.

**تصمیم:** ‏.NET 10 (LTS) با C# 14.

**دلیل:** آخرین LTS، بلوغ بالا، interop عالی با Win32 (SendInput/hook/tray)، انتشار
self-contained و single-file برای پرتابل بودن، اکوسیستم غنی.

**پیامد:** نیاز به .NET 10 SDK برای build. Runtime داخل بسته‌ی self-contained حمل
می‌شود (بدون نیاز کاربر به نصب .NET).

## ADR-0002 — فریم‌ورک UI: WPF + WPF-UI (Fluent)

**وضعیت:** پذیرفته · 2026-07-22

**زمینه:** نیاز به UI مدرن Windows 11 (Fluent، Dark/Light، High-DPI)، پرتابل،
کم‌مصرف، با interop قوی برای SendInput، keyboard hook و System Tray. گزینه‌ها: WPF
+ WPF-UI، WinUI 3، Avalonia.

**تصمیم:** WPF + کتابخانه‌ی WPF-UI.

**دلیل:** WPF کاملاً self-contained و پرتابل publish می‌شود، بالغ است، interop
Win32 بی‌دردسر دارد و RAM پایین. WPF-UI ظاهر Fluent/Win11 می‌دهد. WinUI 3 پرتابل
کردنش دشوار است (packaging، حالت unpackaged ناپایدار) و با اولویت «بدون نصب» تضاد
دارد. Avalonia برای interop ویندوزی کار بیشتری می‌خواهد و مزیت cross-platform آن
فعلاً لازم نیست.

**پیامد:** برخی کنترل‌های خیلی جدید WinUI را نداریم، اما WPF-UI پوشش کافی می‌دهد.
RTL و فارسی در WPF قوی است.

## ADR-0003 — موتور STT هیبرید از روز اول

**وضعیت:** پذیرفته · 2026-07-22

**زمینه:** مالک Faster Whisper را در اولویت گفت، اما «پرتابل بودن» را اولویت اصلی.
Faster Whisper پایه‌ی Python/CTranslate2 دارد که پرتابل‌سازی را سخت می‌کند؛
whisper.cpp کاملاً native است. مالک صراحتاً «هیبرید کامل از روز اول» را انتخاب کرد.

**تصمیم:** هر دو موتور پشت `ISpeechToTextEngine`. بسته‌ی پایه فقط **whisper.cpp**
(via Whisper.net) دارد؛ **Faster Whisper** یک engine pack اختیاری (Python
embeddable + CTranslate2 + IPC ساده) است. انتخاب در تنظیمات؛ fallback خودکار به
whisper.cpp.

**دلیل:** پرتابل بودن پایه تضمین می‌شود و همیشه کار می‌کند؛ کاربران حرفه‌ای می‌توانند
Faster Whisper را برای دقت/سرعت CPU بهتر فعال کنند. اولویت مالک (هر دو موتور) بدون
قربانی‌کردن پرتابلی برآورده می‌شود.

**پیامد:** لایه‌ی adapter و IPC برای sidecar پایتون باید ساخته و تست شود. مدیریت
چرخه‌ی حیات فرآیند sidecar لازم است.

## ADR-0004 — تزریق متن با SendInput، بدون Copy/Paste

**وضعیت:** پذیرفته · 2026-07-22

**زمینه:** متن باید مثل کیبورد واقعی در هر اپ تایپ شود. Copy/Paste ممنوع است.

**تصمیم:** `Pva.Injection` از `SendInput` با `KEYEVENTF_UNICODE` استفاده می‌کند؛
کاراکتربه‌کاراکتر با مدیریت surrogate. کنش‌ها (Enter، Backspace، Undo) به‌صورت کلید
ارسال می‌شوند.

**دلیل:** SendInput استاندارد و سازگار با اکثر اپ‌هاست و Unicode فارسی را پشتیبانی
می‌کند؛ نیازی به کلیپ‌بورد نیست.

**پیامد:** محدودیت UIPI: اگر اپ مقصد elevated باشد و برنامه‌ی ما نباشد، ورودی رد
می‌شود (ریسک R3). با پیام شفاف و راهنمای اجرای elevated مدیریت می‌شود.

## ADR-0005 — ذخیره‌سازی: SQLite + JSON + محافظت secret

**وضعیت:** پذیرفته · 2026-07-22

**زمینه:** نیاز به ذخیره‌ی تاریخچه/یادداشت/متادیتا و تنظیمات، با پرتابل بودن و
رمزنگاری اطلاعات حساس.

**تصمیم:** SQLite (`Microsoft.Data.Sqlite`) برای داده؛ `settings.json` کنار exe
(نسخه‌دار + migration)؛ secretها با DPAPI محلی و گزینه‌ی AES با رمز عبور برای
پرتابلی بین دستگاه‌ها.

**دلیل:** SQLite سبک، بدون سرور و پرتابل است. JSON برای تنظیمات خوانا و قابل‌ویرایش.
DPAPI ساده اما گره‌خورده به دستگاه؛ AES گزینه‌ی انتقال‌پذیر است.

**پیامد:** DPAPI بین ماشین‌ها منتقل نمی‌شود (trade-off مستند در architecture §9).

## ADR-0006 — دامنه‌ی v1: هسته + نوت‌پد داخلی + Sticky Notes

**وضعیت:** پذیرفته · 2026-07-22

**زمینه:** فهرست قابلیت‌ها بسیار بزرگ است. برای «Milestoneهای کوچک» باید مرز v1
مشخص شود. مالک این دامنه را انتخاب کرد.

**تصمیم:** v1 = هسته‌ی دیکته (STT + SendInput + hotkey + دستورات صوتی + میکروفون
شناور + tray + پس‌پردازش فارسی + تنظیمات + بسته‌بندی پرتابل) + نوت‌پد داخلی +
Sticky Notes. بقیه در roadmap.

**دلیل:** یک محصول واقعی و قابل‌فروش با کمترین ریسک و سریع‌ترین مسیر تحویل، سپس
گسترش لایه‌لایه.

**پیامد:** OCR، Clipboard History، Voice Macro، Text Expansion، بازنویسی AI،
marketplace افزونه، MSIX و auto-update بعد از v1.

## ADR-0007 — پس‌پردازش فارسی به‌عنوان ماژول خالص و تست‌محور

**وضعیت:** پذیرفته · 2026-07-22

**زمینه:** خروجی خام Whisper فارسیِ قابل‌قبول نیست (نیم‌فاصله، علائم، اعداد، ترکیب
fa/en). این وجه تمایز محصول است.

**تصمیم:** `Pva.PersianText` یک ماژول خالص و قطعی با تست‌های golden-file. تشخیص با
`initial_prompt` فارسی+فنی و دیکشنری اصطلاحات محافظت‌شده جهت‌دهی می‌شود.

**دلیل:** خلوص و قطعیت، تست‌پذیری و پایداری کیفیت را تضمین می‌کند.

**پیامد:** نیاز به نگه‌داری دیکشنری‌ها و مجموعه‌ی golden رو به رشد.

## ADR-0008 — تضمین آفلاین؛ شبکه فقط opt-in

**وضعیت:** پذیرفته · 2026-07-22

**زمینه:** «کاملاً آفلاین، بدون وابستگی به اینترنت» یک الزام هسته است. اما
قابلیت‌هایی مثل به‌روزرسانی خودکار یا بازنویسی AI ممکن است شبکه بخواهند.

**تصمیم:** کد هسته هیچ فراخوانی شبکه ندارد. هر قابلیت شبکه‌ای (update check،
بازنویسی AI) opt-in، خاموش به‌صورت پیش‌فرض، و کاملاً جدا از هسته است.

**دلیل:** حفظ اعتماد کاربر، حریم خصوصی و تضمین کارکرد آفلاین.

**پیامد:** بازنویسی AI در v1 فقط قاعده‌محور (آفلاین) است؛ نسخه‌ی مبتنی بر LLM بعداً
و opt-in (ریسک R8).

## ADR-0009 — مجوز MIT و مخزن عمومی

**وضعیت:** پذیرفته · 2026-07-22

**زمینه:** مالک خواست پروژه عمومی و تحت MIT روی GitHub منتشر شود.

**تصمیم:** مجوز MIT (فایل `LICENSE`)، مخزن عمومی. کامپوننت‌های شخص ثالث

**دلیل:** انتشار متن‌باز، امکان مشارکت جامعه و سازگاری مجوزی برای استفاده‌ی تجاری.

**پیامد:** رعایت attribution کامپوننت‌ها الزامی است.

## ADR-0010 — پرهیز از FluentAssertions؛ استفاده از xUnit خالص (و در صورت نیاز Shouldly)

**وضعیت:** پذیرفته · 2026-07-22 · جایگزینِ ذکر FluentAssertions در ADRها/مستندات قبلی

**زمینه:** در ابتدا FluentAssertions به‌عنوان کتابخانه‌ی assertion در نظر گرفته شد.
اما FluentAssertions از نسخه‌ی ۸ برای استفاده‌ی تجاری مجوز پولی دارد که با ماهیت
MIT/رایگان پروژه در تضاد است.

**تصمیم:** در تست‌ها از `Assert` خالص xUnit استفاده می‌کنیم؛ در صورت نیاز به سبک
fluent، از **Shouldly** (مجوز MIT) بهره می‌بریم. FluentAssertions اضافه نمی‌شود.

**دلیل:** حذف ریسک مجوزی و هزینه، حفظ سادگی.

**پیامد:** مستندات (`CLAUDE.md`، `architecture.md`) از FluentAssertions به این
تصمیم به‌روزرسانی شدند.

## ADR-0011 — سخت‌گیری کیفیت کد از M0

**وضعیت:** پذیرفته · 2026-07-22

**زمینه:** برای محصول تجاری قابل‌نگهداری، کیفیت باید از ابتدا اجرا شود.

**تصمیم:** `Directory.Build.props` مشترک با `Nullable`، `AnalysisLevel=latest-recommended`،
`EnforceCodeStyleInBuild` و **`TreatWarningsAsErrors` در حالت Release**.
`.editorconfig` قوانین سبک را اعمال می‌کند (namespace فایل‌اسکوپ، modifierها،
مرتب‌سازی usingها). CI روی هر push/PR، Release را build و test می‌کند.

**دلیل:** خطاهای سبک و کیفیت زودتر و به‌صورت خودکار گرفته می‌شوند.

**پیامد:** برخی قوانین برای موارد خاص تنظیم شدند (مثلاً CA1707 در پروژه‌ی تست خاموش،
`require_accessibility_modifiers = for_non_interface_members`).

## ADR-0012 — زبان طراحی: Liquid Glass

**وضعیت:** پذیرفته · 2026-07-22

**زمینه:** مالک هویت بصری مشخصی خواست: آیکون‌های زیبا و غیرمتعارف، انیمیشن و افکت‌های
مدرن، و ظاهر **liquid glass / گلَس‌مورفیسم**.

**تصمیم:** سیستم طراحی «Liquid Glass» با سطوح شیشه‌ی مات، حرکت spring مدرن (تنفس
نور، waveform واکنش‌گر، سوییچ مایع)، آیکون‌های اختصاصی مینیمال، و **ساختار واقعی
برنامه‌ی دسکتاپ** (title bar، nav rail، status bar، میکروفون شناور). پالت **مونوکروم
مدرن و flat** (سبک Linear/Vercel): accent هم‌کنتراست (مشکی در روشن، سفید در تیره)،
بدون گرادیان رنگی — به‌روزرسانی‌شده پس از بازخورد مالک (مینیمال‌تر، flat، مونوکروم).
جزئیات کامل در `docs/design-language.md`؛ پروتوتایپ در `docs/prototypes/liquid-glass.html`.

**دلیل:** تمایز از رقبا، حس محصول ممتاز و تجربه‌ی مدرن مطابق خواسته‌ی مالک.

**پیامد:** پیاده‌سازی در M7 با WPF-UI (Mica/Acrylic) + سطوح شیشه‌ی سفارشی. **گاردریل
کارایی الزامی است:** انیمیشن فقط هنگام فعال بودن، احترام به reduced-motion، و تنزل
graceful روی سخت‌افزار ضعیف تا اهداف «شروع < ۲s، Idle CPU < ۲٪» نقض نشوند (ریسک جدید
R11: تضاد افکت‌های سنگین با کارایی/پرتابلی).

## ADR-0013 — شتاب GPU با Vulkan + سیاست انتخاب مدل بر پایه‌ی benchmark

**وضعیت:** پذیرفته · 2026-07-23

**زمینه:** مالک از کیفیت تشخیص (مدل base) ناراضی بود و آن را با Web Speech API کروم
(بازشناسی ابری گوگل) مقایسه کرد. مدل‌های بزرگ‌تر ویسپر دقت فارسی به‌مراتب بهتری دارند،
اما benchmark واقعی روی سخت‌افزار مرجع (i7-1355U + Iris Xe) نشان داد روی CPU
غیرقابل‌استفاده‌اند: turbo-q5_0 روی CPU ~۱۱x realtime (هر جمله ~۳۶ ثانیه!) چون whisper
هر قطعه را به پنجره‌ی ۳۰ ثانیه‌ای pad می‌کند و هزینه‌ی encoder ثابت است.

**تصمیم:**
1. افزودن `Whisper.net.Runtime.Vulkan` (کنار runtime CPU): در `Auto`/`Gpu` ترتیب
   بارگذاری `[Vulkan, Cpu]` است — Vulkan تقریباً همه‌ی GPUها (حتی iGPU اینتل) را
   پوشش می‌دهد و در نبودش خودکار به CPU برمی‌گردد.
2. beam search به‌جای greedy + استفاده از همه‌ی هسته‌ها منهای یک.
3. **سیاست مدل پیش‌فرض (وقتی مسیر صریح تنظیم نشده):**
   - Auto/Cpu → بهترینِ موجود از `small-q5_1 → small → base` (روی هر دو مسیر امن:
     small-q5_1 با Vulkan ~0.43x و روی CPU ~1.1x).
   - Gpu (انتخاب صریح کاربر) → `turbo-q5_0` اول (کیفیت عالی، ~2.3x با Vulkan).
   - turbo عمداً در حالت Auto انتخاب نمی‌شود تا اگر Vulkan در دسترس نبود، fallback
     به CPU محصول را فلج نکند.
4. پرامپت فارسی نمونه‌محور (Whisper سبک پرامپت را تقلید می‌کند، نه دستور را).
5. پیش‌فرض `fetch-models.ps1` → `small` (فایل `ggml-small-q5_1.bin`)؛ گزینه‌ی `turbo`
   برای کاربران GPU.

**benchmark (jfk.wav، ۹.۹s گفتار، beam، threads=11):**

| مدل | CPU | Vulkan (Iris Xe) |
|---|---|---|
| base | 0.70x | — |
| small-q5_1 | 1.13x | **0.43x** ✅ پیش‌فرض |
| large-v3-turbo-q5_0 | 10.97x ❌ | 2.31x (حالت Gpu) |

**دلیل:** بیشترین کیفیت ممکن که تجربه‌ی دیکته را خراب نکند؛ تصمیم با اندازه‌گیری واقعی
روی سخت‌افزار هدف، نه حدس.

**پیامد:** حجم بسته با runtime Vulkan بیشتر می‌شود؛ مدل small-q5_1 (~۱۹۰MB) مدل توزیع
پیش‌فرض است. اولین رونویسی روی Vulkan ~۱۰s گرم‌کردن (کامپایل shader) دارد — فقط یک‌بار
در هر اجرا. راه آینده: گرم‌کردن غیرمسدودکننده هنگام بارگذاری موتور.
